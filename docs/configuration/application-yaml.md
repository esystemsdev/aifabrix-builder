# application.yaml (application config)

← [Documentation index](../README.md) · [Configuration](README.md)

Generated by `aifabrix create`. Defines your application. The generator transforms nested keys (e.g. `app.key`, `app.displayName`) into the flat structure used by the deployment API and [application-schema.json](../../lib/schema/application-schema.json). **deploymentKey** is not set in application.yaml; it is managed by Controller internally. See [Deployment key](deployment-key.md).

Application config can be **application.yaml**, **application.yml**, or **application.json** on disk; the CLI accepts any of these. Legacy variables.yaml is renamed to application.yaml on first use.

## General parameters (all app types)

### Required for all types

| Field | Description |
| ----- | ----------- |
| `app.key` | Application key identifier (lowercase, numbers, hyphens) |
| `app.displayName` | Display name shown in UI |
| `app.description` | Application description |
| `app.type` | Application type: `webapp`, `functionapp`, `api`, `service`, or `external` |

### Optional (all types)

| Field | Description |
| ----- | ----------- |
| `app.version` | Semantic version (e.g. `1.0.0`). Can be auto-resolved from image. |
| `healthCheck`, `authentication`, `roles`, `permissions` | Runtime and security config |
| `configuration` | Portal UI config (variables with `location`, `portalInput`) |
| `scaling`, `frontDoorRouting` | Deployment and routing config |
| `deployment.environment` | Target environment |
| `repository`, `startupCommand`, `runtimeVersion` | Build and run config |

## Parameters by type: webapp vs external

### Webapp, functionapp, api, service (builder apps)

**Additional required fields:**

| Field | Description |
| ----- | ----------- |
| `port` | Application port (1–65535) |
| `image.name` | Docker image name |
| `image.registry` | Image registry URL |
| `image.registryMode` | Registry mode: `acr`, `external`, or `public` |
| `build.language` | Build language: `typescript` or `python` |

**Additional optional fields:** `build.*` (dockerfile, context, envOutputPath, localPort, containerPort, remoteSyncPath, reloadStart, scripts), `requires.database`, `requires.databases`, `requires.redis`, `requires.storage`, `externalIntegration` (when shipping external systems with the app).

### External apps (type: external)

**Additional required fields:**

| Field | Description |
| ----- | ----------- |
| `externalIntegration` | External system integration block (required). See [External integration and external-system](#external-integration-and-external-system). |

**Not required:** `port`, `image.*`, `image.registryMode`, `build.language` — external apps deploy external systems to the Dataplane, not container images.

**Optional:** `build.*` and other fields when the external integration has custom tooling (e.g. scripts, local validation).

## External integration and external-system

The **externalIntegration** block defines external systems and datasources for pipeline deployment. It is **required** for `type: external` and **optional** for builder apps that ship external systems alongside the application.

### externalIntegration block

| Field | Required | Description |
| ----- | -------- | ----------- |
| `schemaBasePath` | Yes (when block present) | Folder containing system and datasource files |
| `systems` | No | List of external-system filenames (`.json`, `.yaml`, `.yml`). First entry used per application. |
| `dataSources` | No | List of external-datasource filenames |
| `autopublish` | No | If true, pipeline publishes to Dataplane after deployment (default: true) |
| `version` | No | Integration version for schema diffing (e.g. `1.0.0`). Defaults to `1.0.0` when absent. If `app.version` is set, it overrides this value. |

Paths are relative to the application.yaml file or absolute. External system files are validated against `external-system.schema.json`, datasource files against `external-datasource.schema.json` when you run `aifabrix validate <app>`. If `application.yaml` gets out of sync with files on disk, run `aifabrix repair <systemKey>` to fix. Repair also aligns datasource files with the manifest (dimensions and metadataSchema from attributes) when config drifts.

### External system structure

External system files (referenced in `externalIntegration.systems`) follow [external-system.schema.json](../../lib/schema/external-system.schema.json). Main properties:

- **key**, **displayName**, **description**, **type** (`openapi`, `mcp`, `custom`)
- **authentication** – Template-based auth (method, variables, security with `kv://` references)
- **openapi** / **mcp** – Integration bindings (specUrl, serverUrl, etc.)
- **configuration**, **roles**, **permissions** – Optional config and RBAC

External datasource files follow [external-datasource.schema.json](../../lib/schema/external-datasource.schema.json). See [External systems](../external-systems.md) for step-by-step configuration.

### Inline system and dataSources

The application schema supports **inline** system and dataSources for atomic deployment (alternative to file references). The flat deploy manifest can contain a `system` object and `dataSources` array instead of (or in addition to) `externalIntegration.systems` / `.dataSources` file references. Validation: `aifabrix validate <app>`.

## Port and .env resolution

Applies to builder apps (webapp, functionapp, api, service); external apps typically have no port.

| Context | How port is resolved | Example |
| ------- | ------------------- | ------- |
| **Docker Compose** `ports` (host:container) | **Host (first):** `port` (+ developer-id × 100 when dev id > 0). **Container (second):** `build.containerPort` if set and non-empty, else `port`. | `3000:3000` (miso). Keycloak with `port: 8082`, `containerPort: 8080` → `8082:8080`. |
| **Container .env** (e.g. `.env.run` used by the app inside the container) | **With or without `--reload`:** always **container port** — `build.containerPort` if set and non-empty, else `port`. Template var (e.g. `MISO_PORT`) is set to the same value. | `PORT=3000`, `MISO_PORT=3000`. Keycloak: `PORT=8080`. |
| **Local HDD .env** at `build.envOutputPath` (e.g. `../../packages/miso-controller/.env`) | **Without `--reload`:** **localPort** — `build.localPort` when set and > 0, else `port` (+ developer-id × 100 when dev id > 0). Used for host-side tools or other services connecting to the app. | `PORT=3010`, `MISO_PORT=3010`. |
| **Local HDD .env** at `build.envOutputPath` | **With `--reload`:** **same content as container .env** (container port). The file is written so the mounted app directory has the same env as the process inside the container. | Same as container: `PORT=3000`, `MISO_PORT=3000`. |

**Notes:** Compose host port always comes from `port` (not `localPort`), so the mapping is predictable. When `build.containerPort` is empty or unset, `port` is used for the container side.

### Database requirements

When `requires.database` is true, use `requires.databases` to list multiple databases (each with a `name`). Any database whose **name ends with `vector`** is treated as a vector store and gets the **pgvector** extension during db-init when you run the app locally. For Azure or managed Postgres, see [Running: Multiple Databases / Vector stores](../running.md#multiple-databases).

- **`databases[].extensions`** (optional): List of PostgreSQL extension names to create in that database during db-init (e.g. `pgcrypto`, `uuid-ossp`, `vector`, `btree_gin`, `btree_gist`). If the database name ends with `vector`, the `vector` extension is still added automatically if not listed.

## Version and tag

| Field | Purpose |
| ----- | ------- |
| **app.version** | Semantic version (e.g. `1.0.0`). Auto-resolved from image (OCI label `org.opencontainers.image.version` or semver tag) when empty. Defaults to `1.0.0` when no image or no version in image. |
| **image.tag** | Docker image tag (which image to pull). Represents the container artifact. |

When running the image, `builder/<appKey>/application.yaml` is updated with the discovered version. **version** = product/change semantics; **tag** = container artifact.

## Examples

### Minimal webapp

```yaml
app:
  key: myapp
  displayName: "My Application"
  description: "Does amazing things"
  type: webapp
image:
  name: myapp
  registry: myacr.azurecr.io
  registryMode: acr
port: 3000
build:
  language: typescript
```

### Minimal external app

```yaml
app:
  key: hubspot
  displayName: "HubSpot Integration"
  description: "HubSpot CRM integration"
  type: external
externalIntegration:
  schemaBasePath: .
  systems:
    - hubspot-system.yaml
  dataSources:
    - hubspot-datasource-deal.yaml
  autopublish: true
  version: 1.0.0
```

See [env.template](env-template.md), [External systems](../external-systems.md), [Secrets and config](secrets-and-config.md).
