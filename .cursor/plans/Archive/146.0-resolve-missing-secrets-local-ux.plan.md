## Problem (repro: `hubspot-demo`)

From `/workspace/aifabrix-training`:

```text
aifabrix resolve hubspot-demo
âś– Error in resolve command:
   Missing secrets: kv://hubspot-demo/clientSecret
   Secrets file location: â€¦/secrets.local.yaml â€¦
   Run: aifabrix resolve hubspot-demo to generate missing secrets.

â„ą Run "aifabrix doctor" for environment diagnostics.
```

**Root causes:**

1. **Stale `env.template`** â€” Wizard left OAuth lines; `repair --auth bearerToken` updated `*-system.json` to `apikey` + `kv://hubspot-demo/apiKey` but **repair merge keeps orphan lines** not in `buildExpectedByKey` (e.g. active `KV_*_CLIENTSECRET=kv://hubspot-demo/clientSecret` while `clientId` is commented).
2. **Misleading fix hint** â€” Telling the user to run `resolve` again does not help when the missing key is **not required** by the system file (only by a leftover template line).
3. **Wrong resolve environment** â€” [`lib/cli/setup-utility-resolve.js`](../../lib/cli/setup-utility-resolve.js) always passes `'docker'` to `generateEnvFile`, which applies Docker host/port transforms inappropriate for **local integration** resolve (no container).
4. **Doctor footer** â€” [`handleCommandError`](../../lib/utils/cli-utils.js) always appends `aifabrix doctor`, which suggests infra/Docker diagnostics even for a **local secrets** failure.

### Expected user-facing fixes (hubspot-demo)

Either:

- Comment out obsolete OAuth lines in `integration/hubspot-demo/env.template`:

  ```bash
  # KV_HUBSPOT_DEMO_CLIENTSECRET=kv://hubspot-demo/clientSecret
  ```

- Or set only the secret the system file needs:

  ```bash
  aifabrix secret set hubspot-demo/apiKey "<private-app-token>"
  ```

After plan **repair-prune-env**, `aifabrix repair hubspot-demo` should remove or comment stale OAuth `KV_*` lines automatically.
