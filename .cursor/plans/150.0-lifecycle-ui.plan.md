# 150.0 — Builder CLI: Enterprise AI Certification (verify + lifecycle)

## What this plan is

| Repo | Plan | Responsibility |
| --- | --- | --- |
| **aifabrix-dataplane** | [419.0](../../aifabrix-dataplane/.cursor/plans/419.0-integration-lifecycle-validation.plan.md) | Persistence, APIs, dataplane UI cards, governance scenario DB, certification recalculation |
| **aifabrix-builder** (this plan) | **150.0** | CLI commands, product vocabulary, TTY/`--json`, orchestration of existing low-level commands, upload/download scenarios |

**Principle:** Define the **product model first**. If the product model is wrong, the schema will be wrong too. This plan is **not** a database design — it defines what integrators and executives see before dataplane maps it to storage.

---

## The product model (30-second CIO test)

```text
Operational Readiness
+
AI Business Trust
+
Governance Enforcement
=
Enterprise AI Certification
```

| Pillar | Business question | Hero metric |
| --- | --- | --- |
| **Operations** | Does the integration **work**? | **Operational Readiness** (e.g. 94%) |
| **AI Trust** | Will AI **understand** this business data correctly? | **AI Business Context Confidence** (e.g. 92%) |
| **Governance** | Will the **right people** see the **right data**? | **Policy Coverage** (e.g. 100%) |
| **Certification** | Can I **trust this system in production**? | **BRONZE → PLATINUM** + status label |

Nobody cares whether “integration tests passed.” They care whether **it works**, whether **AI understands the business**, and whether **access is correct**.

---

## Object model: system-centric commands

All top-level verify commands take **`<systemKey>`** (external integration), not a datasource key:

```bash
aifabrix verify-operations hubspot
aifabrix verify-trust hubspot
aifabrix verify-governance hubspot
aifabrix lifecycle hubspot
```

**Why:** Demos, management, and CI operate on **systems**. Internally, dataplane may still persist **per-datasource** evidence; the CLI **aggregates** to system scope for output (worst datasource, averages, or documented rollup rules).

**Low-level commands remain** for architects and debug — but **not** at the product top level:

```bash
aifabrix validate
aifabrix test
aifabrix test-integration
aifabrix test-e2e
aifabrix datasource test-e2e <datasourceKey> --debug --verbose
```

---

## Command hierarchy

```text
verify-operations   →  Does the integration work?           →  BRONZE gate
verify-trust        →  Does AI understand the business?     →  SILVER gate
verify-governance   →  Do the right people see the right data? →  GOLD gate
lifecycle           →  Certification report (executive)       →  reads all pillars + tier
```

`lifecycle` is **not** another validator. Default = **certification report only** (GET persisted results). `--run` may execute missing verify steps, then print the report.

---

## 1. `verify-operations <systemKey>`

### Business question

```text
Does the integration work?
```

Not: “Are tests passing?”

### What it verifies (product grouping)

| Group | Checks |
| --- | --- |
| **Connectivity** | Credentials, authentication, authorization |
| **Contracts** | OpenAPI, MCP, mappings |
| **Runtime** | Sync, CRUD, execution |
| **Reliability** | Integration tests, E2E tests (implementation detail) |

### Orchestration (implementation)

1. `aifabrix validate <systemKey>`
2. `aifabrix test <systemKey>`
3. `aifabrix test-integration <systemKey>` (optional `--sync` when flag passed through)
4. `aifabrix test-e2e <systemKey>` (respect existing sync/async flags)
5. Persist rollup via dataplane **`runType=operations`** (419 API) when available; else local-only summary until API ships

**Does not** run trust or governance.

### Primary output (TTY default)

```text
HUBSPOT — OPERATIONS

Operational Readiness
94%

OPERATIONS VERIFIED
```

Or on failure:

```text
OPERATIONS FAILED
```

### Warnings (when other pillars not verified)

```text
⚠ AI trust not verified. Run: aifabrix verify-trust hubspot
⚠ Governance not verified. Run: aifabrix verify-governance hubspot
```

### Details (`--details` or `-d`)

Architect/debug view — **hidden by default**:

```text
Details
────────────────────────────
Connectivity
  Credentials           ✔
  Authentication        ✔
  Authorization         ✔
Contracts
  OpenAPI               ✔
  Mappings              ✔
Runtime
  Sync                  ✔
  CRUD                  ✔
  Execution             ✔
Reliability
  Validation            ✔
  Unit Tests            ✔
  Integration Tests     ✔
  E2E Tests             ✔
```

### Exit code

- `0` — `OPERATIONS VERIFIED`
- `1` — `OPERATIONS FAILED`
- `2` — partial / prerequisites (auth, upload) — align with existing CLI conventions

## 2. `verify-trust <systemKey>`

### Business question

```text
Will AI understand this business data correctly?
```

Not: “Does AI work?”

### What it validates

```text
Business meaning
Business context
Descriptions
Relationships
Labels
Dimensions
```

**Bad datasource (AI can use it but not understand it):**

```text
cust_name, cust_id, attr_1, attr_2
```

**Good datasource:**

```text
Customer Name, Industry, Annual Revenue, Account Owner, Country
```

### Implementation

- Rename **`test-trust`** → **`verify-trust`**; `test-trust` alias warns one release.
- Fan out all datasources under system (same order as today’s `test-trust`).
- Calls dataplane agent metadata validation (404.5); persists **`resultTrust`** via 419 when available.

### Primary output (TTY)

```text
HUBSPOT — AI TRUST

AI Business Context Confidence
92%

AI TRUST VERIFIED
```

Hero metric = **AI Business Context Confidence** — primary differentiator for AI Fabrix.

System rollup: show **worst datasource** confidence when any fail; table in `--details`.

### Exit code

- `0` — all datasources `AI TRUST VERIFIED` (or policy-equivalent trusted decision)
- `1` — any datasource failed / not trusted

## 3. `verify-governance <systemKey>`

### Business question

```text
Will the right people see the right data?
```

### What it checks

```text
RBAC
ABAC
Dimensions
Identity delegation
Filtering
Enforcement scenarios
```

### Implementation

- Rename **`test-governance`** → **`verify-governance`**; alias warns.
- Default: load scenario packs from **dataplane DB** (419 `datasource_protection_scenarios`) after upload — **not** inline `--pack` in product path.
- `--pack` remains **authoring override** only (document in contributor docs).
- Fan out per datasource with uploaded pack; datasources **without pack** → failed (not skipped).

### Primary output (TTY)

```text
HUBSPOT — GOVERNANCE

Policy Coverage
100%

Dimension Coverage
100%

Enforcement Scenarios
15/15 Passed

GOVERNANCE VERIFIED
```

### Exit code

- `0` — `GOVERNANCE VERIFIED` for all in-scope datasources
- `1` — any missing pack, failed scenario, or coverage gap

## 4. `lifecycle <systemKey>`

### Business question

```text
Can I trust this system in production?
```

Not: “What tests ran?”

### Role

**Certification report** — executive summary. **Default = report only** (GET dataplane lifecycle API). **`--run`** = run missing verify steps, then report.

### Primary output (TTY)

```text
HUBSPOT CERTIFICATION REPORT

Certification
────────────────────────
GOLD

Status
────────────────────────
ENTERPRISE AI READY

Operations              ✔ VERIFIED
AI Trust                ✔ VERIFIED
Governance              ✔ VERIFIED
```

When incomplete — **never fake GOLD**:

```text
HUBSPOT CERTIFICATION REPORT

Certification
────────────────────────
BRONZE

Status
────────────────────────
TECHNICALLY READY

Operations              ✔ VERIFIED
AI Trust                — NOT VERIFIED
Governance              — NOT VERIFIED

Recommendations
────────────────────────
2 improvements available
  • Run: aifabrix verify-trust hubspot
  • Run: aifabrix verify-governance hubspot
```

Optional **Recommendations** section: actionable next steps (not generic errors).

### Flags

| Flag | Behavior |
| --- | --- |
| *(default)* | GET `.../external-systems/{key}/lifecycle` — print certification report |
| `--run` | POST `.../lifecycle/run` or sequentially invoke missing verify commands |
| `--json` | Machine-readable envelope (product model fields) |
| `--sync` | Upload local integration (+ scenarios) before verify when used with `--run` |
| `--details` | Include per-datasource breakdown |

## Certification tiers (product)

| Level | Meaning | Requires |
| --- | --- | --- |
| **BRONZE** | Integration works | Operations **VERIFIED** |
| **SILVER** | AI understands business context | BRONZE + AI Trust **VERIFIED** |
| **GOLD** | Access control enforced | SILVER + Governance **VERIFIED** |
| **PLATINUM** | Regulated enterprise ready | GOLD + evidence/audit thresholds (419 dataplane) |

Status labels (examples):

| Tier | Status label |
| --- | --- |
| BRONZE | TECHNICALLY READY |
| SILVER | AI READY |
| GOLD | ENTERPRISE AI READY |
| PLATINUM | REGULATED ENTERPRISE READY |

CLI and dataplane UI must use **the same labels** (419 owns canonical enum; builder maps display strings).

## Expected automated tests

- `verify-operations` invokes validate → test → test-integration → test-e2e in order; stops on first hard fail unless `--continue`
- Default TTY **does not** contain “Integration Tests” unless `--details`

## Manual proof (post-implementation)

Prerequisites: dataplane 419 APIs deployed; `aifabrix auth status` OK.

1. `aifabrix upload <systemKey>` — scenarios included.
2. `aifabrix verify-operations <systemKey>` — Operational Readiness % + OPERATIONS VERIFIED; warnings for trust/governance.
3. `aifabrix verify-trust <systemKey>` — confidence % + business-question line + BUSINESS CONTEXT VERIFIED.
4. `aifabrix verify-governance <systemKey>` — Policy Coverage + Verification Sample + GOVERNANCE VERIFIED.
5. `aifabrix lifecycle <systemKey>` — Last Verified + tier with meaning line; BRONZE until all pillars verified.
6. `aifabrix lifecycle <systemKey> --run` — fills gaps, then GOLD when all pass.
7. `aifabrix verify-operations <systemKey> --details` — shows Validation/Unit/Integration/E2E block.
